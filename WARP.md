**项目**: fafafa.core

**目的**: 规范 TDD 开发流程，避免上下文抖动和"推进过关"

---

## 🎯 核心原则：TDD 优先

### 强制规则：红 → 绿 → 重构

```
1. 红 (Red)    ── 先写失败的测试
2. 绿 (Green)  ── 实现最小代码使测试通过  
3. 重构 (Refactor) ── 优化代码，保持测试通过
```

**违反此规则的行为将导致项目难以持续！**

---

## 🚫 严格禁止的行为

### ❌ 禁止 1: 先写代码，后补测试

```
错误流程：
1. 写实现代码
2. "功能完成了"
3. 想起来要测试
4. 补个测试意思意思

结果：测试质量差，难以发现 Bug
```

### ❌ 禁止 2: 没有测试保护就重构

```
错误流程：
1. 看到"不好的代码"
2. 直接修改
3. "应该没问题吧"
4. 引入 Bug

结果：破坏现有功能，回归困难
```

### ❌ 禁止 3: "推进过关" 心态

```
错误表现：
- "先提交再说，测试以后补"
- "这个测试太难写了，跳过吧"  
- "已经写了很多代码，不想删了"
- "差不多就行了，赶紧推进"

结果：技术债累积，项目腐化
```

### ❌ 禁止 4: 上下文抖动

```
错误表现：
- 做着 A 功能，突然去改 B
- 改着代码，突然去整理 Git
- 重构到一半，又去写新功能

结果：什么都没做完，混乱不堪
```

## ✅ 正确的 TDD 工作流

### 新功能开发

```
1. 理解需求
   ↓
2. 写第一个测试（最简单场景）
   ↓
3. 运行测试 → 应该失败 ✓
   ↓
4. 写最小实现代码
   ↓
5. 运行测试 → 应该通过 ✓
   ↓
6. 提交（代码 + 测试）
   ↓
7. 重复 2-6（添加更多场景）
```

### Bug 修复

```
1. 写重现 Bug 的测试
   ↓
2. 运行测试 → 应该失败（确认 Bug）✓
   ↓
3. 修复代码
   ↓
4. 运行测试 → 应该通过 ✓
   ↓
5. 运行所有测试 → 确保无回归 ✓
   ↓
6. 提交
```

### 重构代码

```
1. 确保有测试覆盖
   ↓
2. 运行所有测试 → 建立基准线 ✓
   ↓
3. 重构代码
   ↓
4. 运行所有测试 → 仍然全部通过 ✓
   ↓
5. 提交
```

## 📝 测试规范

### 测试命名

```pascal
// 模式：Test_<Function>_<Scenario>_<Expected>

procedure Test_ParseISO8601_ValidFormat_Success;
procedure Test_ParseISO8601_InvalidFormat_ReturnsFalse;
procedure Test_DurationAdd_Overflow_Saturates;
```

### 测试内容（3 个必须）

```pascal
// 1. 正常路径（Happy Path）
procedure Test_FromMs_NormalValue_Success;

// 2. 边界条件（Boundary）
procedure Test_FromMs_MaxValue_Saturates;
procedure Test_FromMs_MinValue_Saturates;
procedure Test_FromMs_Zero_Success;

// 3. 异常情况（Exception）
procedure Test_FromMs_Overflow_Throws;
```

### 测试结构（AAA 模式）

```pascal
procedure Test_DurationAdd_PositiveValues_CorrectSum;
begin
  // Arrange - 准备
  var D1 := TDuration.FromMs(100);
  var D2 := TDuration.FromMs(200);
  
  // Act - 执行
  var Result := D1 + D2;
  
  // Assert - 断言
  AssertEquals(300, Result.AsMs);
end;
```

## 🔄 小步迭代原则

### 每次只做一件事

```
✅ 好的节奏：
- 写一个测试
- 实现一个功能
- 提交一次
- 重复

❌ 坏的节奏：
- 写 10 个功能
- "差不多了"
- 大批量提交
```

### 提交频率

```
理想：每 15-30 分钟提交一次
最多：不超过 1 小时
最少：每个功能必须提交

每次提交应该：
✓ 可编译
✓ 测试通过
✓ 功能完整（即使很小）
```

## 🎓 Warp AI 协作规范

### AI 应该做的

1. **提醒 TDD**
   ```
   用户："实现 ISO 8601 解析"
   AI："我们先写测试。最简单的情况是什么？"
   ```

2. **测试优先展示**
   ```
   AI 响应顺序：
   1. 展示测试代码
   2. 运行测试（失败）
   3. 展示实现代码
   4. 运行测试（通过）
   5. 提交
   ```

3. **防止上下文抖动**
   ```
   用户："顺便也改一下 X"
   AI："我们先完成当前的 Y，提交后再处理 X"
   ```

4. **阻止推进过关**
   ```
   AI：检测到测试失败
   → "测试没通过，我们先修复再继续"
   → 不要建议"跳过"或"以后再说"
   ```

### AI 不应该做的

1. ❌ 不生成没有测试的代码
2. ❌ 不建议"测试以后补"
3. ❌ 不在测试失败时继续推进
4. ❌ 不建议"因为简单所以不用测试"
5. ❌ 不一次性修改大量文件

## 📊 质量检查清单

### 每次提交前必检

- [ ] 所有新功能有测试
- [ ] 所有测试通过
- [ ] 没有注释掉的测试
- [ ] 提交信息清晰
- [ ] 编译无警告

### 代码审查要点

- [ ] 测试覆盖正常、边界、异常
- [ ] 代码遵循命名约定
- [ ] 无重复代码
- [ ] 热路径使用 inline
- [ ] 接口设计一致

## 💡 最佳实践

### 1. 测试驱动思维

```
看到需求 → 思考如何测试 → 写测试 → 写代码

NOT

看到需求 → 写代码 → 可能测试？
```

### 2. 删除的勇气

```
代码不好？删除重来！
设计有问题？推倒重构！
测试覆盖保护你，不要怕。
```

### 3. 持续验证

```
每写一点代码 → 运行测试
每改一点代码 → 运行测试
每重构一次 → 运行测试
```

### 4. 一次聚焦一事

```
当前任务：实现 ISO 8601 解析
聚焦：只做这一件事
完成：提交
下一任务：实现 RFC 3339 解析
```

## 🎯 Warp AI 标准响应模板

### 用户要求实现功能时

```
我们用 TDD 方式实现：

第一步：写测试（最简单的情况）
[展示测试代码]

第二步：运行测试（应该失败）
[运行命令]

第三步：实现最小代码
[展示实现代码]

第四步：运行测试（应该通过）
[运行命令]

第五步：提交
[git commit]

完成一个场景后，我们再添加下一个场景的测试。
```

### 用户想要跳过测试时

```
我理解你可能觉得这个测试不重要，但是：

1. TDD 是项目规范（见 WARP.md）
2. 没有测试的代码将来很难维护
3. 写测试通常会发现设计问题

让我们先写一个最简单的测试，不会花很多时间。
```

### 检测到上下文抖动时

```
注意：我们当前正在做 X，你现在提到了 Y。

为了避免上下文抖动，建议：
1. 先完成 X（预计还需 N 步）
2. 提交 X  
3. 再开始 Y

你想继续 X 还是切换到 Y？
```

### TDD 的价值

✅ **质量** - 更少 Bug，更早发现问题  
✅ **设计** - 可测试的代码通常是好设计  
✅ **文档** - 测试即可执行的文档  
✅ **信心** - 敢于重构，敢于改进  
✅ **速度** - 长期看更快（少 Debug）
