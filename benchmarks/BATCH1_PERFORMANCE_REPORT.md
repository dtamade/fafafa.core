## 3. Event (事件) 性能测试

### 测试场景

| 场景 | 类型 | 说明 |
|------|------|------|
| AutoReset | 自动重置 | 每次信号只唤醒一个等待者 |
| ManualReset | 手动重置 | 每次信号唤醒所有等待者 |

### 性能数据

#### 自动重置事件 (Auto-Reset)
```
 2 线程: 2,768,640 ops | 1,536,876 ops/sec | 650.67 ns/op
 4 线程: 1,130,496 ops |   941,110 ops/sec | 1,062.57 ns/op
 8 线程:   694,528 ops |   630,737 ops/sec | 1,585.45 ns/op
16 线程:   506,368 ops |   388,788 ops/sec | 2,572.10 ns/op
```

#### 手动重置事件 (Manual-Reset)
```
 2 线程: 1,048,320 ops | 1,047,387 ops/sec | 954.76 ns/op
 4 线程: 2,362,880 ops | 2,360,332 ops/sec | 423.67 ns/op
 8 线程: 1,851,136 ops | 1,681,017 ops/sec | 594.88 ns/op
16 线程: 1,786,880 ops | 1,621,261 ops/sec | 616.80 ns/op
```

4. **使用场景**: 手动重置更适合广播通知场景

## 4. CondVar (条件变量) 性能测试 ⚠️

### 测试状态: **发现严重问题**

#### 成功的测试
```
1P1C (1 生产者 / 1 消费者): 1,850,368 ops | 1,848,677 ops/sec | 540.94 ns/op
```

#### 失败的测试
```
2P2C (2 生产者 / 2 消费者): 致命错误
```

### 错误详情

**错误类型**: glibc 断言失败
```
pthread_mutex_cond_lock.c:96: __pthread_mutex_cond_lock_adjust: 
Assertion `mutex->__data.__owner == 0' failed.
```

### 问题分析

1. **根本原因**: CondVar 实现存在线程安全问题
2. **触发条件**: 多生产者/多消费者场景 (2P2C 及以上)
3. **影响范围**: 所有多线程 CondVar 使用场景
4. **严重程度**: **CRITICAL** - 导致程序崩溃

### 建议措施

1. **立即行动**: 
   - 暂停 CondVar 在生产环境的使用
   - 深入调查 Unix 平台 CondVar 实现
   - 检查 mutex 所有权管理逻辑

2. **后续工作**:
   - 修复线程安全问题
   - 增加多线程压力测试
   - 验证 Windows 平台是否存在相同问题
