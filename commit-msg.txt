Title
simd(cpuinfo/diagnostics/dispatch): honor OS enablement; add backend priority; improve core-count fallback; docs Raw vs Usable

Body
Summary
- x86: make SIMD usability depend on OS enablement (OSXSAVE/XCR0), not just hardware support
- diagnostics: print OSXSAVE/XCR0 and Raw vs Usable, add grep-friendly summary lines
- dispatch: introduce explicit backend priority order (AVX512 > AVX2 > SSE2 > NEON > RISCVV > Scalar)
- windows: add GetActiveProcessorCount(ALL_PROCESSOR_GROUPS) fallback to improve logical CPU count
- docs: describe capability model (Raw vs Usable), clarify 256-bit backend = AVX2+ on x86
- tests: add invariants ensuring GenericUsable ⊆ GenericRaw, 512 ⇒ 256, and backend alignment

Details
- cpuinfo (x86)
  - OSXSAVE from CPUID leaf1 ECX bit27; read XCR0 only when OSXSAVE=1
  - Build GenericRaw (hardware) and GenericUsable (OS enabled) sets
  - HasFeature uses GenericUsable; GetSupportedBackends only exposes OS-usable backends
- diagnostics
  - Print OSXSAVE and XCR0
  - Show Raw/Usable for SIMD-128/256/512, AES, FMA, SHA
  - Add summary lines: "Summary: SIMD-256 raw=true usable=false (OSXSAVE=?, XCR0=$...)"
  - First-detection time measured after ResetCPUInfo (reflects real init cost)
- dispatch
  - Explicit priority order independent of registration enumeration
  - Select highest priority backend that is both registered and available
- windows core count
  - Prefer GetActiveProcessorCount(ALL_PROCESSOR_GROUPS) when available; fallback to SYSTEM_INFO
- tests
  - Add TestOSEnablementInvariants: Usable ⊆ Raw; 512 ⇒ 256; backends reflect usable features
- docs
  - Add "Capability model (Raw vs Usable)" to README_SIMD.md; clarify 256-bit backend = AVX2+

Rationale
- Avoids false positives where CPU advertises AVX/AVX-512 but OS context is not enabled
- Makes dispatch choice stable and predictable across builds
- Improves diagnosability and portability (Windows processor groups)
- Guards correctness with invariants in tests

Notes
- Consider Linux-only core-count summary and optional startup log of chosen backend
- If demand arises for AVX-only backend, introduce sbAVX with tests later

Breaking changes
- None in public API; backend exposure aligns with OS usability (may change reported backends on systems where OS hasn't enabled AVX/AVX-512)
