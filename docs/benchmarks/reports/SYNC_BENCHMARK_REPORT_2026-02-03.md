## 关键发现

### 1. Mutex 性能优秀
- **单线程**: ~24ns/op (41M ops/sec) - 非常快
- **多线程**: 竞争下仍保持 ~10M ops/sec

### 2. RWLock NoReentry 模式是性能最佳选择
- **NoReentry 模式**: 119ns/op (8.4M ops/sec)
- **默认模式**: 1,466ns/op (0.7M ops/sec)
- **性能提升**: 约 **12 倍**

`★ Insight ─────────────────────────────────────`
- NoReentry 模式跳过了线程 ID 检查和重入跟踪，直接使用快速路径
- 如果你的代码不需要递归锁定，强烈建议使用 NoReentry 模式
- ReaderBias 选项对性能影响不大（仅 ~2%）
`─────────────────────────────────────────────────`

### 3. Guard 开销合理
- Read Guard 开销: ~185ns（相对于直接调用）
- Write Guard 开销: ~190ns
- RAII 安全性带来的开销是可接受的

### 4. Named 同步原语性能
- NamedMutex 比进程内 Mutex 慢约 **6 倍**（152ns vs 24ns）
- 这是因为需要跨进程协调的系统调用开销
- 创建/销毁开销较大（~37ms 创建 1000 个）

## 性能建议

1. **低延迟场景**: 使用 `IMutex`（24ns/op）
2. **读多写少**: 使用 `IRWLock` + `NoReentry` 模式（119ns/op 读）
3. **跨进程**: 使用 `INamedMutex`，但要复用实例，避免频繁创建
4. **异常安全**: 使用 Guard 包装器，开销可接受
