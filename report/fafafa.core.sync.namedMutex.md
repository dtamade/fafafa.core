## 遇到的问题与解决方案

### 🔧 技术挑战

1. **平台差异处理**
   - **问题**：Windows 和 Unix 的命名同步原语差异很大
   - **解决**：Windows 使用原生互斥锁，Unix 最终采用文件锁实现
   - **结果**：实现了统一接口下的平台最优性能和稳定性

2. **命名规范统一**
   - **问题**：不同平台的命名规则差异（Windows 路径 vs Unix 文件名）
   - **解决**：在各平台实现中进行名称验证和转换
   - **结果**：用户可以使用简单名称，底层自动处理平台差异

3. **Unix 平台实现重构**
   - **问题**：POSIX named semaphore 导致访问违例和线程安全问题
   - **解决**：完全重写为基于文件锁的实现
   - **结果**：消除了所有访问违例，提高了稳定性和可靠性

### 🏗️ 设计决策

1. **接口继承层次**
   - **决策**：`INamedMutex` 继承自 `IMutex` 而不是直接继承 `ILock`
   - **原因**：保持与现有 mutex 模块的一致性，复用 `GetHandle` 方法
   - **优势**：用户可以将命名互斥锁当作普通互斥锁使用

2. **Unix 平台实现重新选择**
   - **初始决策**：使用 POSIX named semaphore
   - **问题发现**：导致访问违例、线程安全问题和资源管理复杂性
   - **最终决策**：改用文件锁 (fcntl) 实现
   - **优势**：更稳定、更简单、自动清理、无访问违例

3. **接口设计优化**
   - **问题**：原始设计包含状态设置方法，破坏了无状态原则
   - **解决**：移除 `SetTimeout/GetTimeout`，改为参数化的超时方法
   - **结果**：更清晰的接口设计，避免了线程安全问题

4. **工厂函数设计**
   - **决策**：提供多个便利函数而不是单一的复杂构造函数
   - **原因**：提高易用性，支持不同的使用场景
   - **结果**：用户可以根据需要选择最合适的创建方式

## 代码审查与重构

### 🔍 重大代码审查发现

1. **严重的线程安全问题**
   - **问题**：Unix 实现中的 `FIsLocked` 字段在多线程环境下不安全
   - **影响**：可能导致访问违例和竞态条件
   - **解决**：完全移除线程状态跟踪，依赖系统级锁机制

2. **接口设计不一致**
   - **问题**：`SetTimeout/GetTimeout` 方法破坏了无状态设计原则
   - **影响**：增加了复杂性和潜在的线程安全问题
   - **解决**：重新设计为无状态的参数化超时方法

3. **平台实现稳定性问题**
   - **问题**：POSIX named semaphore 导致访问违例
   - **影响**：测试失败，生产环境不可用
   - **解决**：完全重写为基于文件锁的实现

### 🔧 重构成果

1. **Unix 平台完全重写**
   - 从 POSIX named semaphore 改为文件锁 (fcntl)
   - 消除了所有访问违例错误
   - 简化了资源管理逻辑
   - 提高了稳定性和可靠性

2. **接口优化**
   - 统一了方法命名：`IsOwner` → `IsCreator`
   - 移除了有状态的超时设置
   - 添加了参数化的超时方法
   - 保持了跨平台一致性

3. **测试修复**
   - 修复了基于错误假设的测试逻辑
   - 适应了文件锁的行为特性
   - 确保了测试的稳定性和可靠性
